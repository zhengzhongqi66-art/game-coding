# 风险与决策

## 1. 当前建议确认的关键决策

### 决策 1：先把项目定义为空间构筑项目

原因：

- 这是当前所有讨论中真正稳定的共同核心
- 可以避免过早被单一皮肤锁死

### 决策 2：以修仙作为第一个垂直切片主题

原因：

- 与现有策划积累衔接最好
- 能自然承载供能、朝向和五行逻辑
- 对当前目标用户有较高识别度

### 决策 3：项目结构以“中枢 + 主题世界”推进

原因：

- 能提供长期结构
- 能解释内容多样性
- 能降低原创大世界搭建压力

### 决策 4：Meta 保持轻量

原因：

- 核心乐趣必须来自构筑与适应
- 过重成长会掩盖核心玩法问题

### 决策 5：推迟 PvP

原因：

- 平衡成本太高
- 首个 Demo 应优先证明 PvE 闭环成立

## 2. 主要制作风险

### 风险 A：项目变成普通背包玩法换皮

应对建议：

- 强化供能回路
- 强化朝向规则
- 强化世界规则差异
- 让修仙机制真正进入空间层

### 风险 B：战斗结算不可读

应对建议：

- 尽早制作战斗日志
- 尽早明确触发时序
- 高亮激活来源
- 原型期控制视觉复杂度

### 风险 C：题材野心跑得比产能更快

应对建议：

- 先只做一个世界
- 严格采用小世界包
- 在验证前不要并行启动多个大方向

### 风险 D：系统数据失控

应对建议：

- 尽早确定物品 schema
- 统一触发时机命名
- 尽量自动校验数据格式

### 风险 E：中枢空间膨胀

应对建议：

- 让中枢以准备功能为主
- 避免过早加入大地图或社交层
- 让它先服务节奏和决策，而不是服务“内容堆叠”

## 3. 需要在原型后复盘的问题

- 核心空间载体最终应该表现为背包、纳戒、阵图还是其他形式？
- 对外宣传时应更强调修仙题材、轮回世界，还是纯系统卖点？
- 场地规则变化做到什么程度不会伤害可读性？
- Demo 后是否值得加入异步镜像对抗？
- 是否需要一个固定主角身份来增强承载？

## 4. 后续功能决策规则

任何新功能都应先回答以下问题：

1. 它是否提升了空间构筑决策？
2. 它是否强化了中枢 -> 世界 -> 中枢的主循环？
3. 它是否提升了 build 开奖的可读性？
4. 它是否适合小团队产能？
5. 它是否能帮助首个 Demo 更成立？

如果大多数答案是否定的，就应推迟。
