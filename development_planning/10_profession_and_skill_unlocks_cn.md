# 职业与技能解锁

## 1. 文档目的

这份文档用于定义首个制作版本中，职业、职业技能以及长期解锁结构应如何运作。

目标是让职业选择同时具备：

- 可见性
- 机制意义
- 主题表达
- 小团队可控的制作范围

## 2. 职业系统核心规则

职业应被视为一组“预设但可定向投入”的成长身份。

核心规则：

- 玩家不会被永久锁死在单一幻想身份中
- 职业技能在主神空间中由玩家主动购买
- 系统应允许玩家按偏好定向发展，而不是完全随机分配职业

这样既保留了长期成长的灵活性，也保留了职业的作者设计感。

## 3. 第一次解锁时刻

第一次购买职业技能，是一个重要的身份门槛。

当玩家第一次购买职业技能后：

- 职业身份会被明确激活
- 玩家携带容器的表现形式会发生变化
- 战斗空间会开始获得职业相关的身份变化
- 职业相关的表现层开始出现
- 职业相关规则可以进入 run 层

这个时刻应该像一次“身份承诺”，而不是一笔普通数值购买。

## 4. 职业表现层

每个职业至少应在以下层面产生可见变化：

- 携带容器形式
- 战斗空间表现或结构提示
- UI 身份提示
- 技能图标语言
- 在适当情况下的道具亲和提示

例如：

- 偏修仙的职业不应继续使用普通背包，而应改成 `纳戒`

这个视觉变化应立刻传达出：

- “我的角色逻辑变了”
- “我的构筑空间开始属于某种职业幻想”

## 5. 首版职业系统范围

首个制作版本中的职业输出应保持紧凑。

建议的最小职业包：

- 一个身份概念
- 一种容器或背包表现形式
- 一条战斗空间变体规则或表现变化
- 一条核心被动规则
- 一小条技能分支
- 一种道具亲和偏向

不要一开始就做：

- 多个同深度职业并行展开
- 大型天赋树
- 职业专属整包内容

## 6. 技能购买模型

职业技能应在主神空间中购买。

建议的购买规则：

- 玩家使用专门的职业解锁资源来购买技能，具体资源名后续再正式定义
- 技能购买应是主动决策，而不是随机获得
- 玩家在购买前应始终清楚知道技能会改变什么

首版应优先强调：

- 文案清晰
- 选项数量少
- 收益可见

## 7. 职业技能类别

建议的首版职业技能类别：

- 战斗空间修正
- 储物空间修正
- 商店或经济偏向
- 道具家族亲和
- 一条职业专属规则变化

这样能让职业系统始终贴着核心 loop，而不是漂成抽象 RPG 数值层。

## 8. 与构筑循环的连接

职业技能应强化主游戏循环：

- 准备
- 摆放
- 适应
- 验证

好的职业技能应影响这些问题：

- 当前可用的战斗空间有多少
- 职业解锁后战斗空间呈现出什么结构或形态变化
- 哪类支撑件更容易发挥价值
- 储物空间如何影响 run 内规划
- 哪类奖励或商店项更值得追求

不好的职业技能是那些：

- 只加泛用数值
- 不改变决策
- 让所有 build 越来越像

## 9. 职业特设

每个职业最终都应拥有至少一个职业特设。

建议的首版规则：

- 每个职业只做一个能用一句话讲清楚的职业特设

可接受的例子：

- 修仙职业更容易驱动有朝向要求的灵器
- scavenger 风格职业拥有一个和储物空间有关的转换技巧
- ritual 风格职业拥有一条特殊支撑链接规则

首版要证明的是“玩法手感不同”，不是“内容数量很多”。

## 10. 与道具和世界设计的关系

职业身份应影响玩家侧逻辑，但不应完全取代世界身份。

设计规则：

- 世界定义外部环境条件
- 职业定义玩家侧偏向

这个区分很重要，因为：

- 世界多样性负责遭遇新鲜感
- 职业身份负责玩家归属感

## 11. UX 要求

职业系统必须把这些信息做得足够明显：

- 当前激活的是哪个职业身份
- 已解锁了哪些技能
- 下一步可以购买哪些技能
- 最新一次购买后发生了什么变化
- 当前携带容器发生了什么改变
- 战斗空间在职业激活后发生了什么变化

如果玩家买了职业技能，下一局却几乎感受不到变化，那这个系统就是低效的。

## 12. 验证问题

只有当以下问题大多能回答“是”时，这套系统才算成立：

- 第一次技能购买是否有记忆点？
- 玩家是否能解释自己当前职业在做什么？
- 职业是否同时改变了表现层和决策层？
- 玩家是否会想继续朝偏好的职业方向投入？
- 职业身份是否没有压过基础构筑系统本身？

## 13. 对制作的直接影响

这套规划后续应继续补充：

- 首版职业名单
- 职业容器表现表
- 技能解锁资源定义
- 首批职业技能表
- 职业与道具亲和表
- 职业 UI 状态规格
