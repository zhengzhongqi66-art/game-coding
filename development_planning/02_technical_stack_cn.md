# 技术选型

## 1. 引擎建议

### 推荐引擎：Godot 4.x

推荐原因：

- 更适合小团队快速原型
- 对 2D、UI 与系统驱动项目更友好
- 适合格子逻辑、构筑界面与战斗表现的快速迭代
- 项目负担相对较轻
- 更利于小体量独游的长期维护

### 为什么 Godot 尤其适合本项目

本项目高度依赖：

- 格子或有限空间逻辑
- 数据驱动的物品系统
- UI 占比高的构筑交互
- 可重复调试的战斗模拟
- 快速试错与快速改规则

这些都与 Godot 的优势较匹配。

### 什么情况下 Unity 才值得考虑

只有在后续明确转向以下目标时，Unity 才更有必要：

- 大量依赖成熟第三方商业工具链
- 团队扩张后强依赖 Unity 招聘池
- 明确接入更多中大型中间件

对于当前范围，这些都不是优先需求。

## 2. 编程语言建议

### 推荐：GDScript 为主

适合用于：

- 玩法逻辑
- 物品编排逻辑
- 战斗流程控制
- UI 状态管理
- 内容工具脚本

推荐理由：

- 原型迭代速度快
- 更适合设计驱动型系统开发
- 做编辑器工具与调试工具成本低

### 补充建议：仅在确认热点后局部引入 C#

不建议一开始就上 C#。

只有在性能分析明确显示瓶颈后，再考虑用于：

- 高频模拟计算
- 大规模规则推演
- 重型程序化计算

## 3. 项目目录建议

推荐的顶层目录结构：

- `scenes/`
- `scripts/`
- `data/`
- `art/`
- `audio/`
- `ui/`
- `tools/`
- `tests/`
- `docs/`

推荐的脚本模块拆分：

- `scripts/core/`
- `scripts/run/`
- `scripts/hub/`
- `scripts/combat/`
- `scripts/items/`
- `scripts/worlds/`
- `scripts/meta/`
- `scripts/ui/`
- `scripts/debug/`

## 4. 数据驱动架构

项目应尽量采用数据驱动方式组织内容。

建议外置的数据包括：

- 物品定义
- 敌人定义
- 世界修正规则
- 事件池
- 奖励表
- Run 参数

推荐格式：

- 原型阶段优先 JSON
- 稳定后可逐步迁移到 Godot `Resource`

建议原则：

- 不稳定内容优先轻量文本化
- 稳定内容再逐步转为编辑器资产

## 5. 核心技术架构

### 玩法架构

建议采用规则优先结构：

- 物品数据负责定义静态身份
- 效果组件负责定义行为片段
- 模拟系统按固定阶段结算
- 视觉层监听结果而不是持有游戏真相

这有助于内容规模扩大后保持清晰。

### 模拟架构

战斗应尽量保持可复现、可调试的确定性结算：

- 战前准备阶段
- 开场触发阶段
- 周期性行动阶段
- 被动反应阶段
- 死亡与收尾阶段
- 奖励结算阶段

这样更有利于：

- 调试
- 回放
- 平衡测试

### UI 架构

UI 应作为状态表现层，而不是规则持有层。

主要状态包括：

- 空间格子状态
- 物品选中状态
- 悬停检查状态
- 战斗信息状态
- 奖励和商店状态

避免把关键逻辑塞进 UI 组件内部。

## 6. 存档与持久化

建议拆分多个保存层：

- 设置存档
- 玩家进度存档
- 当前 Run 快照
- 调试或回放快照

最少需要保存：

- 已解锁内容
- 用户选项
- 当前进行中的 Run
- 教程完成状态
- 核心统计数据

## 7. 工具与协作流程

推荐工具：

- Git 版本控制
- GitHub 或 Gitea 协作
- Figma 做 UI 线框
- Aseprite 做像素物品和图标
- Krita 或 Photoshop 做概念和宣传图
- Markdown 文档持续记录设计

## 8. 测试策略

测试重点应放在系统正确性，而不只是人工试玩。

建议包括：

- 物品效果结算测试
- 战斗顺序回归测试
- 数据合法性校验
- 固定种子复现
- 平衡沙盒模式

最重要的早期调试功能包括：

- 强制添加物品
- 强制载入种子
- 直接跳到指定节点
- 查看触发链
- 导出战斗日志

## 9. 构建目标

首发目标平台：

- Windows PC

后续可考虑：

- Steam Deck 验证

在首个公开 Demo 稳定前，不建议扩大平台范围。
