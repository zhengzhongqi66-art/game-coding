# 项目立项书（中文版）

## 一、项目概述

### 1. 项目暂定名

暂定为一款以“空间构筑”为核心的独立游戏项目，当前可使用的内部描述为：

**空间构筑 + 自动战斗 + 轮回副本 + 轻 meta 的小体量独立游戏。**

当前不建议过早锁定最终正式名称，原因是项目母结构已经相对清晰，但题材包装仍保留一定调整空间。

### 2. 项目一句话定义

玩家需要在一个有限空间内安排自己的 build，通过摆放、邻接、朝向、供能和联动来决定战斗表现；随后进入不同主题的副本世界进行验证，获得奖励后回到中枢空间继续调整构筑，并不断推进更深层内容。

### 3. 当前阶段定位

本项目当前处于立项前期到原型规划阶段。

现阶段目标不是直接展开全量制作，而是先完成：

- 项目母结构确认
- 技术路线确认
- 首个垂直切片范围确认
- 小团队可执行的开发路径确认

### 4. 项目目标

本项目的阶段性目标分为三层：

#### 第一层：验证玩法成立

验证以下核心问题：

- 玩家是否会因为空间摆放本身而产生乐趣
- 玩家是否能从物品联动中获得持续的新鲜感
- 自动战斗是否能够清晰、直观地“开奖”
- 中枢空间与副本轮回是否能形成明确目标感

#### 第二层：验证产品表达成立

验证以下产品层问题：

- 项目是否能被一句话讲清楚
- 玩家是否能在短时间内理解玩法卖点
- 首个世界主题是否足够形成记忆点
- 是否具备做 Steam Demo 的基本潜力

#### 第三层：验证小团队落地成立

验证以下制作层问题：

- 小团队是否能在有限周期内做出可玩原型
- 玩法系统是否适合低成本扩展
- 内容生产方式是否足够模块化
- 题材扩展是否不会拖垮制作节奏

## 二、项目核心判断

### 1. 项目真正的核心不是“修仙”，而是“空间构筑”

目前最重要的判断是：

项目不应被理解为“纯修仙题材游戏”，也不应被简单理解为“背包游戏”。

项目真正的核心是：

**在有限空间中安排 build，并让空间结构直接影响战斗结果。**

因此，背包、纳戒、阵图、轮回匣、灵台等，都可以只是表现形式；真正不可替代的是以下玩法基础：

- 空间有限
- 物品形状不同
- 摆放位置有差异
- 邻接关系会触发联动
- 朝向会改变效果
- 某些物品负责供能
- 某些物品负责消耗条件并爆发

### 2. 自动战斗不是核心趣味本体，而是验证器

本项目不应把“自动战斗”误判为核心玩法本身。

自动战斗的真正作用是：

**把玩家刚刚完成的空间决策和 build 组合，用一种可视化、可理解的方式结算出来。**

因此，战斗承担的是“开奖”功能，而不是主要操作乐趣来源。

玩家的乐趣主要来自三个连续环节：

- 排布：这个东西该摆哪里
- 组合：哪些东西会彼此激活
- 开奖：这套 build 到底是怎么赢的

### 3. 项目母结构应为“中枢空间 + 轮回副本”

与普通单线关卡推进相比，“中枢空间 + 轮回副本”的结构更适合本项目，原因包括：

- 能自然解释为什么不断进入不同副本
- 能自然解释为什么每轮战斗后返回准备区
- 能自然承载世界切换和题材扩展
- 能以较低文本成本支撑轻叙事与目标感

项目主循环应明确为：

**中枢空间调整构筑 -> 进入副本世界验证 -> 获得奖励 -> 回到中枢继续调整 -> 进入下一轮**

### 4. 修仙不是唯一母题，但仍是当前最强首发方向

当前判断并不是放弃修仙，而是重新给修仙定位。

更合理的理解是：

- 修仙可以作为第一批最适合落地的世界主题
- 修仙可以作为首批最成熟的流派与视觉抓手
- 修仙可以作为首个垂直切片的核心表达
- 但整个项目不一定永久锁死在单一修仙世界

这意味着：

**修仙是当前最强的起手方向，但项目结构本身应保留继续扩展到其他主题世界的能力。**

## 三、目标用户与产品定位

### 1. 目标用户

项目第一阶段更适合面向以下用户：

- 中文区独立游戏玩家
- 喜欢研究 build、联动和组合的玩家
- 喜欢“系统乐趣”胜过长文本叙事的玩家
- 愿意尝试新机制、小体量、有辨识度独游的玩家

### 2. 产品气质

项目整体气质建议为：

- 规则清楚
- 节奏紧凑
- 系统驱动
- 爽点明确
- 轻松有梗

其中“有梗”应主要体现在：

- 道具命名
- 联动表现
- 世界主题反差
- 敌人与事件设计
- 结算反馈语气

而不应建立在大量段子文案或重文本表达上。

### 3. 市场定位

本项目当前更适合定位为：

**中文小圈层甜品型独立游戏，先验证机制成立，再判断是否扩大题材和市场表达。**

现阶段不建议直接按“大体量全球化商业项目”思路规划。

## 四、核心玩法结构

### 1. 核心玩法循环

玩家的标准循环应为：

1. 回到中枢空间
2. 查看奖励与资源变化
3. 购买、替换或调整物品
4. 重新安排空间构筑
5. 选择下一个副本世界或路线
6. 进入战斗、事件、奖励、精英和 Boss 节点
7. 结算本轮结果
8. 返回中枢空间继续推进

### 2. 空间构筑层

这是项目最核心的玩法层。

玩家需要在有限空间里处理以下决策：

- 大件强力物品是否值得占位
- 小件补缝物品是否能补足关键联动
- 某个核心输出是否必须转向
- 某个供能件是否需要贴近目标
- 某个连锁件是否需要保持线路完整

如果空间构筑设计成功，那么玩家会对“移动一格、转一次、换一个邻接关系”产生明显感知。

### 3. 战斗层

战斗建议继续采用自动结算为主的设计方向。

战斗不强调高强度即时操作，而强调：

- build 是否启动成功
- 联动是否足够有观赏性
- 输赢原因是否足够清楚
- 玩家是否能自然想到“下一把怎么改”

### 4. 副本层

副本当前更适合采用轻量结构，而非大型开放探索。

首个版本建议采用：

- 小地图或节点推进
- 战斗节点
- 事件节点
- 奖励节点
- 精英节点
- Boss 节点

副本的本质是：

**带有主题规则的验证场。**

### 5. 中枢空间层

中枢空间承担的作用包括：

- 作为每轮结束后的回归点
- 提供商店、解锁或成长接口
- 提供路线选择与风险判断
- 提供轻量叙事承载
- 形成“上一轮发生了什么，下一轮准备去哪里”的节奏感

中枢空间不应在首版中膨胀为大型经营系统。

### 6. 轻 meta 层

项目可接受轻度长期成长，但不应过重。

首版适合保留的 meta 内容包括：

- 新物品解锁
- 新事件解锁
- 新副本修正解锁
- 商店条目扩展
- 路线分支解锁

不建议首版做大量永久属性膨胀，以免掩盖核心玩法是否真的成立。

## 五、机制关键词与差异化方向

### 1. 当前最值得保留的机制关键词

以下关键词可视为项目的核心资产候选：

- 空间构筑
- 自动战斗
- build 爆发
- 邻接规则
- 朝向规则
- 供能 / 充能
- 条件触发
- 意外组合
- 副本轮回
- 中枢空间
- 轻 meta

### 2. 修仙方向下最值得保留的差异化

如果首个世界继续采用修仙方向，那么最值得坚持的不是“仙侠外皮”，而是两类真正能进入玩法层的规则：

#### 五行进入空间规则

五行不应只体现在背景设定或战斗克制上，而应直接进入摆放逻辑，例如：

- 相生关系带来供能或增幅
- 相克关系带来削弱或副作用
- 特定组合触发额外路线

这样五行才能从设定词变成可感知的玩法语言。

#### 法器朝向与灵气回路

例如：

- 飞剑强调朝向
- 丹炉强调包围
- 灵石强调供能
- 符箓强调连线

这能使项目的空间语言形成独特辨识度，避免变成同类产品的简单换皮。

## 六、技术选型建议

### 1. 引擎建议

建议使用：

**Godot 4.x**

理由如下：

- 更适合小团队快速原型
- 更适合 2D、UI、格子和系统驱动型项目
- 更适合快速迭代空间摆放和战斗反馈
- 对小体量独游而言项目负担较轻
- 后续做内部工具和调试支持也更灵活

### 2. 编程语言建议

建议以：

**GDScript 为主**

用于：

- 核心玩法逻辑
- UI 流程
- 战斗结算编排
- 内容工具
- 数据校验

只有在后续明确出现性能瓶颈时，才考虑局部引入 C#。

### 3. 技术架构建议

项目应尽量采用数据驱动结构。

建议优先外置的内容包括：

- 物品定义
- 敌人定义
- 事件定义
- 奖励池
- 世界规则
- 副本参数

这样更有利于：

- 快速试错
- 后续扩内容
- 平衡调整
- 避免逻辑散落在界面脚本里

### 4. 平台建议

首个明确目标平台建议为：

- Windows PC

后续可再考虑：

- Steam Deck 适配

当前不建议在项目早期分散到更多平台。

## 七、功能模块拆分

### 1. 运行主流程模块

负责：

- 开始一轮
- 进入中枢
- 进入副本
- 节点推进
- Boss 结算
- Run 结束

### 2. 空间构筑模块

负责：

- 格子或空间逻辑
- 物品摆放、旋转、移动
- 占位检测
- 邻接检测
- 朝向检测
- 联动预览
- 无效摆放提示

这是最核心的系统模块。

### 3. 物品系统模块

负责：

- 物品静态数据
- 物品触发条件
- 物品效果定义
- 标签、稀有度、流派归属
- 可复用行为组件

### 4. 供能与连线模块

负责：

- 能量生成
- 能量传递
- 路径或连线逻辑
- 激活判定
- 断链提示

这一模块很可能成为项目差异化的关键。

### 5. 战斗结算模块

负责：

- 战斗初始化
- 行动顺序
- 触发处理
- 伤害、护盾、治疗、状态处理
- 召唤或附属单位处理
- 输赢判定

### 6. 奖励与经济模块

负责：

- 战后奖励
- 商店生成
- 刷新与价格逻辑
- 世界主题奖励倾向
- 支线风险收益

### 7. 世界主题模块

每个世界建议作为一个紧凑内容包来制作，至少包含：

- 视觉主题
- 敌人池
- 一个场地规则
- 一个奖励方向
- 一个精英组
- 一个 Boss

### 8. 中枢空间模块

负责：

- 准备阶段空间
- 商店或解锁入口
- 世界选择
- 风险选择
- 轻叙事承载

### 9. meta 进度模块

负责：

- 解锁逻辑
- 长期记录
- 轻成长内容

### 10. 可读性与调试模块

必须尽早建设，包括：

- 触发说明
- 联动高亮
- 战斗日志
- 种子复现
- 强制加物品
- 快速跳关
- 胜率与使用率统计

## 八、内容生产策略

### 1. 世界内容采用“小包体取样”策略

为了避免项目内容失控，每个世界建议只固定生产以下内容：

- 3 个视觉识别符号
- 2 个敌人机制
- 1 个场地规则
- 1 套主题奖励
- 1 个支线事件钩子

这有助于控制范围，同时保持主题辨识度。

### 2. 内容生产顺序

建议所有新世界都按以下顺序制作：

1. 先定规则
2. 再定奖励
3. 再定敌人表现
4. 再定视觉符号
5. 最后补事件和包装

这样能避免出现“美术很好看，但玩法没有落点”的问题。

### 3. 首个世界建议

首个世界建议仍然采用：

**修仙秘境 / 试炼世界**

原因包括：

- 与当前已有策划衔接最好
- 五行、法器、灵气等概念天然适合转成空间规则
- 对当前目标用户识别成本较低
- 容易形成第一版视觉和玩法记忆点

## 九、开发节奏与里程碑建议

### 1. 阶段目标

项目当前最合理的近期目标不是“直接做完整游戏”，而是：

**做出一个足以证明核心玩法成立的垂直切片。**

### 2. 建议阶段划分

#### 阶段一：预制作锁定

完成内容：

- 产品定义统一
- 技术栈确定
- 首个世界确定
- 模块拆分确定
- 首版边界确定

#### 阶段二：核心交互原型

完成内容：

- 格子摆放
- 旋转与占位
- 邻接判定
- 朝向预览
- 简单激活预览

目标是先证明“摆东西本身就有趣”。

#### 阶段三：战斗验证原型

完成内容：

- 战斗场景
- 基础敌人
- 行动结算
- 伤害和状态处理
- 基础战斗反馈

目标是证明“构筑结果能被清晰开奖”。

#### 阶段四：完整 Run 闭环

完成内容：

- 中枢空间
- 副本节点流
- 奖励与商店
- 精英
- Boss

目标是证明完整循环成立。

#### 阶段五：首个垂直切片

完成内容：

- 修仙世界包装
- 首批物品池
- 五行摆放规则
- 灵气供能逻辑
- 首个 Boss
- 基础引导

目标是形成可展示版本。

### 3. 小团队推荐周期

如果团队保持聚焦，建议以 12 至 16 周为一个较现实的原型到切片周期。

## 十、当前明确不建议首版做的内容

为了保证项目不失控，当前不建议在首版直接纳入以下内容：

- 联网 PvP
- 大量章节剧情
- 多角色并行开局
- 多个完整世界同时上线
- 大型经营或村庄系统
- 重文本叙事驱动
- 大规模外围系统堆叠

这些内容不一定永远不做，但不应该进入首个生产目标。

## 十一、主要风险与对应建议

### 1. 风险：变成“普通背包游戏换皮”

建议：

- 强化供能与连线逻辑
- 强化朝向与空间关系
- 强化世界规则差异
- 让修仙规则真正进入摆放层而不是停留在设定层

### 2. 风险：战斗结果看不懂

建议：

- 提前做战斗日志
- 提前做触发顺序可视化
- 提前做联动来源标记
- 初期减少过度炫技特效

### 3. 风险：题材扩展太快导致制作失控

建议：

- 先只做一个世界
- 严格使用小包体世界结构
- 没有验证之前不要平行开多个大方向

### 4. 风险：系统定义不统一导致后期难维护

建议：

- 提前统一物品 schema
- 提前统一触发时机命名
- 提前统一数据结构和内容模板

---

## 十二、当前建议立项结论

综合现有讨论与收敛结果，当前最建议的立项结论为：

1. 将项目正式定义为“空间构筑驱动的独立游戏项目”。
2. 以“自动战斗 + 中枢空间 + 轮回副本 + 轻 meta”作为母结构。
3. 将“修仙秘境 / 修仙流派”作为首个垂直切片方向，而不是整个项目唯一永久题材。
4. 技术上以 `Godot 4 + GDScript` 作为首选方案。
5. 目标先锁定小团队可完成的可玩原型与垂直切片，不追求首版大而全。

这意味着本项目当前已经具备进入原型前准备阶段的条件。

接下来的关键，不再是继续发散题材，而是：

**把首个可玩版本的内容边界、物品池、规则层级和 UI 结构进一步写实、写细。**
