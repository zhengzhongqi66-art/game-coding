# Risks And Decisions

## 1. Current recommended decisions

### Decision 1: Treat the game as a space-construction project first

Reason:

- this is the true shared core across all current discussions
- it prevents overcommitting to one skin too early

### Decision 2: Use xianxia as the first vertical slice theme

Reason:

- best bridge from current concept work
- easy to express through items, energy, and orientation rules
- strong recognition value for the current target audience

### Decision 3: Build around hub plus themed worlds

Reason:

- gives long-term structure
- justifies varied content
- reduces worldbuilding burden

### Decision 4: Keep meta progression light

Reason:

- the main fun must come from arrangement and adaptation
- heavy meta can hide a weak core loop

### Decision 5: Delay PvP

Reason:

- balance cost is too high early on
- first demo should prove the PvE loop

## 2. Main production risks

### Risk A: The game becomes "just another backpack game"

Mitigation:

- emphasize energy routing
- emphasize direction rules
- emphasize world rule variation
- make first-world mechanics meaningfully different from genre defaults

### Risk B: Combat becomes unreadable

Mitigation:

- build combat logs early
- use clear effect timing
- highlight activation sources
- keep first prototype visuals simple
- require post-battle cause summaries before adding heavy visual polish

### Risk C: Theme ambition outruns production capacity

Mitigation:

- ship one world first
- use small world packages
- avoid starting multiple world themes in production at once

### Risk D: Systems become data chaos

Mitigation:

- define item schema early
- standardize triggers and timing names
- validate data automatically where possible
- track item strategic role in addition to raw content count

### Risk E: The hub becomes bloated

Mitigation:

- keep the hub functional
- avoid adding large navigation or social layers
- use it as a preparation and reward space first

### Risk F: The game feels theoretically interesting but not replayable

Mitigation:

- track whether players change layouts voluntarily
- test whether rewards create real branching strategy
- review whether failures suggest a clear next adjustment
- avoid adding content that does not improve decision density

## 3. Open questions to revisit after prototype

- Should the core space be presented as backpack, ring, formation board, or another thematic container?
- Should the first public-facing pitch lean more on xianxia, on looped worlds, or on pure systems language?
- How much stage rule variation can players handle before readability suffers?
- Is asynchronous mirror combat worth adding after the demo?
- Does the game need one fixed protagonist identity, or can the hub framing stay lighter?

## 4. Decision rules for future changes

Any future feature should be checked against these questions:

1. Does it improve space-construction decisions?
2. Does it improve the hub -> world -> hub loop?
3. Does it improve readability of build payoff?
4. Can it fit inside a small-team production plan?
5. Does it strengthen the first demo?

If the answer is mostly no, the feature should be postponed.
