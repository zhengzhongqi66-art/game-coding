# 可玩性验证

## 1. 文档目的

这份文档用于定义团队应当如何判断：这个游戏是否正在变得真正“好玩”，而不只是概念上看起来有潜力。

它主要回答四个对制作至关重要的问题：

1. 空间摆放是否持续产生有意义的决策？
2. 战斗反馈是否清晰、可读、且有满足感？
3. 完整一轮 run 是否足够驱动玩家继续？
4. 失败是否能帮助玩家学习，而不是显得随机或委屈？

这份文档应作为路线图之上的“验证层”，而不是替代路线图本身。

## 2. 验证原则

项目不能因为以下原因就被判定为“已经好玩”：

- 题材看起来有吸引力
- 道具想法纸面上很巧妙
- 战斗特效足够花哨
- 测试者能讲清楚玩法概念

只有当玩家反复表现出以下行为时，项目才算真的在证明自己：

- 主动、有目的地调整布局
- 能预判至少一部分结果变化
- 能理解构筑为什么成功或失败
- 看完结果后立刻想再试一个调整方案

## 3. 核心可玩性问题

### A. 空间决策测试

只有当玩家经常面对以下真实取舍时，构筑层才算成立：

- 空间效率
- 相邻收益
- 朝向对齐
- 能量路径
- 支撑依赖
- 未来灵活性与当前强度的取舍

验证标准：

- 挪动一格应当经常产生有意义的预期变化
- 旋转道具应该经常重要到足以被作为核心规则教学
- 支撑型道具应该像“引擎零件”，而不是纯粹的税

失败信号：

- 玩家主要按数值高低摆放
- 玩家只要摆出一个“能用”的布局就停止调整
- 朝向和相邻机制被遗忘，而不是被主动利用

### B. 战斗可读性测试

只有当战斗能清楚揭示构筑逻辑时，战斗层才算成立。

玩家应该能回答：

- 什么被触发了
- 什么没能触发
- 主要伤害来源是什么
- 主要防御崩溃点是什么
- 哪一次摆放选择最关键

验证标准：

- 在胜利或失败之后，大多数测试者无需设计师解释，就能说出两到三个决定性原因

失败信号：

- 玩家把结果描述为随机
- 玩家无法把强力连锁追溯回摆放选择
- 失败来得突然，且让人觉得不公平

### C. Run 驱动力测试

只有当 run 循环能持续制造短期和中期动力时，整体 loop 才算成立。

玩家应该感受到：

- 对下一个奖励有即时好奇
- 愿意围绕不完美选择做适应
- 对精英或 boss 检定有期待
- 下一轮想尝试不同构筑思路

验证标准：

- 一次较短的完整 run 应至少产生一次“我想换个方式试试”的瞬间

失败信号：

- 奖励显得平、无差异、可替换
- 遭遇流程像构筑界面之间的填充物
- 前几场战斗之后，玩家就不再关心后面会发生什么

### D. 公平失败测试

只有当失败能带来学习时，失败才是可接受的。

一次好的失败应该帮助玩家识别出：

- 一个错误的摆放
- 一个缺失的支撑条件
- 一个过于贪心的奖励选择
- 一个没有尊重的世界规则

验证标准：

- 玩家在输掉后的一分钟内，应能看出一个较明确的下次调整方向

失败信号：

- 玩家把失败完全归因于发牌
- 玩家无法判断自己的构筑究竟是“差一点成”还是“根本不行”
- run 很早就注定失败，而且几乎没有补救机会

## 4. 原型阶段应跟踪的指标

这些指标应在内部测试和早期外部测试中持续跟踪。

### 构筑指标

- 每次奖励步骤后的平均布局调整次数
- 一局 run 中至少被旋转过一次的道具占比
- 已摆放但未激活或支撑不足的道具占比
- 构筑时间与战斗时间的平均比值

### 战斗指标

- 平均战斗时长
- 最高伤害来源出现频率
- 触发失败次数
- 能量断供出现频率
- run 结束是死于爆发还是死于慢性崩溃

### Run 指标

- 平均 run 长度
- 失败前到达的遭遇深度
- 奖励选择分布
- reroll 使用率
- boss 到达率

### 学习指标

- 能解释上一场失败原因的测试者占比
- 能在下一场前说出构筑目标的测试者占比
- 会主动说“再来一局”的测试者占比

## 5. Playtest 检查表

每轮有意义的原型测试后，都应使用这份清单复盘。

### 构筑

- 玩家是否在没有被强迫的情况下频繁调整布局？
- 是否至少有一个摆放决策带来了“意外但能理解”的结果？
- 支撑型道具是否显得有策略价值？

### 战斗

- 玩家是否能跟上主要触发顺序？
- 玩家是否能指出某次连锁为何成功？
- 玩家是否能指出防御为何失效？

### 奖励与经济

- 奖励是否真实地分化了策略路线？
- 玩家是否遇到过“即时强度”和“未来灵活性”的取舍？
- 商店或奖励界面是否促进了适应，而不是自动选最优？

### 驱动力

- 玩家是否想看到下一个遭遇或 boss？
- 玩家是否会为下一轮形成新的构筑假设？
- 世界规则是否真正影响决策，而不是只停留在背景层？

### 失败与恢复

- 最后一次失败是否讲得清楚？
- 在奖励不顺时，run 中是否至少给过一次翻盘或补救机会？
- 玩家是否对结果保有“这是我自己打出来的”归因感？

## 6. 各里程碑最低验收门槛

### 里程碑 1：核心交互原型

必须证明：

- 摆放变化经常会改变预期价值
- 相邻和朝向规则是可理解的
- 玩家会自发优化布局

如果以下问题仍存在，就不应推进：

- 构筑只是战斗前的繁琐准备

### 里程碑 2：战斗验证原型

必须证明：

- 战斗按可读顺序结算
- 玩家能指出主要胜负原因
- 道具触发能被清楚检查

如果以下问题仍存在，就不应推进：

- 战斗看起来很热闹，但因果关系不清楚

### 里程碑 3：完整 run 循环原型

必须证明：

- 奖励选择会引发真实的布局改动
- 遭遇顺序能制造张力
- 一次短 run 能促使玩家产生适应和再试动机

如果以下问题仍存在，就不应推进：

- 整个 run 像一组彼此割裂的界面

### 里程碑 4：首个世界 vertical slice

必须证明：

- 第一世界会改变玩家的构筑方式
- 主题规则能形成有记忆点的策略身份
- 世界在视觉和机制上都可辨识

如果以下问题仍存在，就不应推进：

- 世界基本只是通用系统上的一层皮

### 里程碑 5：Demo 候选版本

必须证明：

- 新玩家在一次 session 内就能理解核心 loop
- 大多数玩家无需额外讲解即可完成教学
- 在 demo 条件下，失败依然是可学习的

如果以下问题仍存在，就不应推进：

- 只有设计师在旁边解释时，游戏才显得好玩

## 7. 对现有计划的直接影响

这套规划需要显式支持以下内容：

- 战斗可读性 UI 必须被具体定义，而不是停留在暗示层
- 经济和奖励分支需要更强的定义
- 失败复盘工具需要尽早存在
- 第一世界的机制身份必须比“主题皮肤”更锋利
- 道具池规划应跟踪策略角色，而不仅是道具数量

## 8. 决策规则

当多个优先级发生冲突时，应按以下顺序保护：

1. 有意义的构筑决策
2. 可读的战斗因果关系
3. 有驱动力的奖励适应
4. 公平失败与再来一局的欲望
5. 表现层打磨

如果表现层提升了观感，但削弱了可读性，那么应优先保留可读性。
