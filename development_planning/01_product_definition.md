# Product Definition

## 1. Working project statement

The project is a small-scope indie game with this backbone:

- space construction
- auto-battle payoff
- looped dungeon worlds
- hub return and rebuild
- light meta progression

The player loop is:

1. Return to the hub.
2. Review rewards and new options.
3. Rearrange the limited build space.
4. Choose a themed world or route.
5. Enter encounters, events, elites, and boss content.
6. Watch the build resolve through combat.
7. Return to the hub and adapt.

This means the project should be planned as a "build validation loop" game, not as a story-first RPG and not as a pure backpack-only game.

## 2. Design pillars

### Pillar A: Space arrangement is the real gameplay

The player should care about:

- where an item is placed
- what it touches
- which direction it faces
- whether it consumes too much room
- which support pieces enable the main engine

If moving one tile does not matter, the core design is failing.

### Pillar B: Combat is a readable reveal, not the main skill test

Combat exists to show:

- whether the build actually works
- why it wins or loses
- what should be changed next run

The player fantasy is not high-action control. It is "I built this, and now I see it come alive."

### Pillar C: Hub and world rotation create long-term motivation

The game should feel like:

- prepare in the hub
- enter a world with a theme
- survive or exploit its rules
- return stronger
- aim for the next depth step

This is what gives the project a sustainable roguelite structure.

### Pillar D: Theme sampling beats heavy original worldbuilding

The game should not depend on a single giant lore setting.

Instead:

- keep a stable hub frame
- sample recognizable world themes
- translate theme rules into combat and space mechanics

This keeps content production lighter and makes world variety easier.

## 3. What the game is not

The first production target is not:

- a text-heavy narrative game
- a large overworld exploration RPG
- a real-time action combat game
- a multiplayer-first autobattler
- a content-heavy "many modes at launch" project

## 4. Theme position

Xianxia remains highly valuable, but its role is:

- first major world theme
- first mature item family
- first strong visual hook
- first identity anchor for Chinese players

It should not lock the entire project unless later playtests prove that a single-theme product is stronger.

## 5. Target player

Primary early audience:

- Chinese-speaking indie players
- players who enjoy build experimentation
- players who like system-driven fun more than long dialogue
- players who appreciate slightly playful tone and recognizable motifs

## 6. First playable target

The first serious playable version should prove:

1. Rearranging limited space is fun by itself.
2. Item interactions create surprising build outcomes.
3. Auto-battle outcomes are readable and satisfying.
4. The hub -> world -> hub loop gives enough forward motivation.
5. A first world theme can carry visual and mechanical identity without heavy writing.

## 7. First release boundary

Should include:

- one hub
- one complete run loop
- one world theme with strong mechanical identity
- a limited but expressive item pool
- one boss path
- light permanent unlocks
- basic run history and build readability tools

Should not include yet:

- full online PvP
- multiple complete world packs
- large branching story campaign
- many playable characters
- complex social systems
- heavy town/base simulation
